🔄 Facility Registration Workflow

1 Facility Owner Submits Registration
Public form: facility name, contact info, address, optional subdomain request
2 Super Admin Reviews Request
Registration appears in super admin panel for review
3 Super Admin Sets Up Facility
Creates facility record, sets up initial branch, configures basic settings
4 Super Admin Customizes Branding
Uploads logo, sets color scheme (primary, secondary, accent)
5 Super Admin Creates Owner Account
Creates facility owner user with configurable role (admin/manager/etc.)
6 Owner Receives Credentials
Welcome email with login credentials and access instructions
7 Owner Accesses Their Panel
Logs in via /admin (context-aware) or subdomain.facility.com with custom branding

🛣️ Routing Strategy

Path-Based Routing

URL: yourapp.com/admin

Facility context is determined by:

  • User's facility_id from database
  • Automatic data filtering by facility
  • Custom branding loaded from facility record

Subdomain Routing

URL: evergreen.yourapp.com/admin

Facility detection via:

  • Subdomain extraction from request
  • Lookup facility by subdomain field
  • Set facility context automatically

📅 Implementation Phases

Phase 1: Database & Models

  • Add customization fields to facilities table
  • Add facility_id to users table
  • Create facility_registrations table
  • Update Facility and User models
  • Migrate existing data to "Evergreen Oasis Care Home"

Phase 2: Multi-Tenancy Core

  • Create facility context middleware
  • Implement global scopes for data isolation
  • Add subdomain detection support
  • Update all models with facility relationships

Phase 3: Super Admin Panel

  • Create FacilityRegistrationResource
  • Build facility customization interface
  • Implement registration approval workflow
  • Add role assignment during setup

Phase 4: Registration System

  • Create public registration form
  • Build registration controller
  • Add email notifications
  • Create success/confirmation pages

Phase 5: Dynamic Branding

  • Update AdminPanelProvider for dynamic branding
  • Implement logo loading from facility
  • Apply custom color schemes
  • Test branding across facilities

🔒 Security & Data Isolation

Data Isolation Mechanisms

  • Global Scopes: Automatically filter all queries by facility_id
  • Middleware Protection: Prevent cross-facility access at request level
  • Relationship Scoping: Ensure related data respects facility boundaries
  • Access Control: Super admins can only manage registrations, not access facility data
  • Subdomain Validation: Verify subdomain matches user's facility
